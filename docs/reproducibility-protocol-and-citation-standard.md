# Reproducibility Protocol and Citation Standard

## Purpose

This document defines the operating standard for reproducibility, evidence handling, and source attribution in the `bay-area-startup-survival-engine` repository. The objective is not only to preserve computational repeatability, but also to ensure that every empirical claim can be reconstructed from public files that are legally acquired by the repository owner and kept outside the public Git history.

The repository is intentionally structured around a strict separation between: (i) analytical text and code that may be published, and (ii) raw public data files that are referenced, documented, and consumed within the acquisition layer, but not redistributed through GitHub. This design is consistent with a conservative interpretation of data stewardship and prevents accidental republication of third-party statistical files in a derivative repository.

## Scope

The protocol applies to all Markdown research notes, empirical appendices, Python scripts, derived CSV outputs, and any future notebooks or figures that may be added to the project. The standard covers five distinct layers:

1. **Source identification**: each raw file must have an institutional owner, source page, direct download path when available, and date of access.
2. **Acquisition discipline**: raw files are downloaded manually or programmatically into `data/raw/`, the non-versioned acquisition layer excluded from version control.
3. **Transformation discipline**: every script that mutates raw data into a standardized panel must write auditable outputs to `data/processed/` or `outputs/`.
4. **Analytical traceability**: every load-bearing number in Markdown should be traceable either to an official source or to a named derived file produced by a repository script.
5. **Publication discipline**: no raw public file is committed to the public repository unless the owner explicitly decides to do so and the relevant terms permit redistribution.

## Repository-level reproducibility principles

### 1. Deterministic folder logic

The project should preserve a stable directory grammar:

- `data/raw/` for source files held in the non-versioned acquisition layer and excluded from Git.
- `data/processed/` for normalized machine-readable intermediates.
- `outputs/` for model-ready panels, reporting tables, and figure inputs.
- `src/` for executable pipeline scripts.
- `analysis/` and `docs/` for formal Markdown documents.

This folder logic materially reduces hidden state. A reviewer should be able to understand whether a number is raw, transformed, or interpretive merely from the path convention.

### 2. Source-first computation

Every transformation must begin from a documented institutional file rather than from manually edited spreadsheets. This matters because public micro-aggregates can be revised; institutional revisions should supersede ad hoc edits. The Business Formation Statistics methodology explicitly notes that actual and projected formation series are revised as new administrative information becomes available, which means reproducibility requires preserving the file provenance, not only the end result [1].

### 3. Explicit treatment of modular replication environments

The repository must remain operational when only part of the raw-data universe is available within the acquisition layer. This is already necessary in practice: Business Formation Statistics files are compact and easy to transport, whereas QCEW area extracts can be extremely large because the program publishes county-level and industry-level information across the entire United States [2]. Therefore, scripts should:

- fail hard only on genuinely required files;
- warn, not crash, when optional families are missing;
- write coverage diagnostics that make missing geography or year windows visible.

### 4. Derived outputs must be reproducible, not authoritative

Any CSV or Markdown file generated by repository scripts is an analytical artifact, not an official source. It can be convenient for replication, but it is not the ground truth. Official institutional sources retain primacy whenever revisions occur.

## Citation standard

### A. In-text citation rule

Each specific numerical claim, dataset definition, coverage rule, or methodological distinction should have an immediate citation in square brackets, for example `[1]` or `[2][3]`. Where a statement reflects the author's own calculations from a downloaded public file, the citation should point to the source dataset and the derived-file lineage should be described in prose.

### B. Reference format

References should follow a compact institutional style:

`[n] Institution — Short title, year, type, URL`

Examples:

- `[1] U.S. Census Bureau — Business Formation Statistics, February 2026 release, monthly release/PDF, <...>`
- `[2] U.S. Bureau of Labor Statistics — Guide to QCEW Data Sources, 2026, technical documentation, <...>`

### C. Dual citation for transformed claims

When a claim depends on both an official source and a repository transformation, references should indicate both layers:

- institutional dataset page or release;
- script name or derived file name.

Example:

`Bay Area business applications declined by 3.6% in 2024 on annual county totals reconstructed from the Census annual county workbook and aggregated in the repository's county panel [3][4].`

Where:
- `[3]` is the Census county annual workbook;
- `[4]` is the repository transformation script or derived CSV.

## Minimum metadata required for each raw source

Every raw source used in the repository should be documented with the following fields:

| Field | Required | Description |
|---|---:|---|
| Institution | Yes | Legal/institutional owner of the source |
| Dataset family | Yes | BFS, BDS, QCEW, SUSB, CBSA delineation, etc. |
| Source page | Yes | Landing page containing context and metadata |
| Direct file path | Preferred | Direct downloadable file URL when available |
| Local filename | Yes | Filename expected in `data/raw/` |
| Access date | Yes | Date on which the file was obtained |
| Frequency | Yes | Monthly, annual, county-level, etc. |
| Coverage notes | Yes | Geography, years, exclusions, revision rules |
| Redistribution posture | Yes | Whether the raw file is excluded from GitHub |

## Why raw files are excluded from public Git

There are four reasons.

First, exclusion avoids transforming a research repository into an unofficial mirror of federal statistical files. Second, it keeps the repository light and reviewable. Third, it reduces the risk of silently mixing revisions from different download dates. Fourth, it preserves a clean legal boundary between the author's analytical work and the institutions' statistical property.

The BLS explicitly notes that the QCEW publication dataset is vastly larger than most BLS products and that the full data are best accessed through downloadable files rather than the publication database [2]. This makes a non-versioned raw-data layer both operationally sensible and methodologically cleaner.

## Reproducibility checklist before publication

Before pushing new analytical material, confirm the following:

1. Raw files are present in the acquisition layer and excluded from Git.
2. Source filenames match script expectations exactly.
3. Transformation scripts run from a clean environment without manual intervention.
4. Outputs are written from code, not hand-edited.
5. Every important numerical statement in Markdown has a reference.
6. Any estimate, bridge, or proxy is clearly labeled as such.
7. Coverage gaps are disclosed explicitly.
8. The distinction between observed values and model-implied values is never blurred.

## Recommended citation language for public-facing Markdown

When a result is computed from repository transformations of public data, use language such as:

- “Author calculations from U.S. Census Bureau Business Formation Statistics county annual applications file and monthly state release.”
- “Derived from BLS QCEW raw area extracts using the repository transformation pipeline.”
- “Computed from Census CBSA delineation files matched to county GEOIDs.”

This formulation is conservative and transparent. It informs the reader that the figure is analytical, while preserving the institutional source chain.
